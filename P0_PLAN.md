## P0-1. Утечка BOT_TOKEN и секретов в репозитории

Файл: ENV_INSTRUCTIONS.md (токен, ADMIN_ID); история git (файл уже закоммичен).

Почему это ошибка:
Токен бота — полный доступ к API Telegram от имени бота. Любой, кто видит репозиторий (или историю), может управлять ботом, читать/писать сообщения, публиковать в каналы, куда бот добавлен. ADMIN_ID — идентификатор человека, лишняя персональная привязка в открытом виде.

Как исправить:
Вне репозитория (вручную владельцем):
1) у @BotFather — /revoke или новый токен;
2) новый токен только в Railway/хостинге как secret env, не в git.

В репозитории (после подтверждения на правки):
- заменить ENV_INSTRUCTIONS.md на шаблон без реальных значений (например BOT_TOKEN=your_token_here);
- при желании добавить .env.example с теми же плейсхолдерами;
- опционально: git filter-repo / BFG, если репо уже светилось публично — иначе достаточно revoke (старый токен всё равно мёртв).

Последствия:
Старый токен перестанет работать — обязательно обновить env на Railway. Бот с новым токеном снова работает. Риск захвата бота по утёкшему токену снимается. История git может ещё содержать старый токен — после revoke он бесполезен.

От вас: подтвердите, что сами отзовёте токен; в плане правок репо — только обезличивание ENV_INSTRUCTIONS.md (и при согласии .env.example).

## P0-2. Старт без проверки BOT_TOKEN

Файл: main.py (примерно строки 16–18, 68: TOKEN = os.getenv(...) → Bot(token=TOKEN)).

Почему это ошибка:
При отсутствии или пустом BOT_TOKEN бот падает с неочевидной ошибкой aiogram/Telegram или ломается при первом запросе. Нет fail-fast и понятного сообщения для деплоя.

Как исправить:
Сразу после чтения env: если not TOKEN (None / пустая строка) — logging.error(...) и sys.exit(1) с текстом вроде: «Задайте BOT_TOKEN в переменных окружения». Не создавать Bot без валидного токена. Опционально: предупреждение, если нет CHANNEL_ID (не критично — есть default).

Последствия:
Локально/на Railway без env процесс сразу завершится с ясной причиной. С корректным env поведение не меняется. Чуть проще отладка деплоя.

## P0-3. Подписки на направления не доставляют уведомления

Файлы:
- main.py — UI/хендлеры подписки («Подписка на направления», driver_subscribe_*), публикация груза (logistician_add_cargo_contact, confirm_single_msg_cargo);
- db.py — add_subscription, get_subscribers_for_route (есть), вызовов уведомления нет.

Почему это ошибка:
Пользователю обещают: «будете получать уведомления о новых грузах», но после add_cargo / публикации в канал get_subscribers_for_route нигде не вызывается. Функция мёртвая с точки зрения продукта — ложное ожидание и потеря доверия.

Как исправить (вариант A — довести фичу, рекомендуется):
После успешного add_cargo (оба сценария: пошаговый и «одним сообщением») для (origin, destination):
1) telegram_ids = get_subscribers_for_route(origin, destination);
2) каждому — bot.send_message с кратким текстом груза (маршрут, вес/тип, цена по выбранной политике, контакт по политике P0-4);
3) ошибки по одному пользователю (блок, удалил чат) — try/except + log, остальные получают;
4) защита: driver_id is None при подписке — отказ с текстом «сначала выберите роль водителя»;
5) по возможности не слать автору груза, если он же подписан (если когда-то будет dual-role).

Альтернатива B: убрать кнопку и хендлеры подписки + неиспользуемые функции в db.py, пока нет рассылки.

Для MVP логистики вариант A предпочтительнее.

Последствия (A):
Подписки начинают работать на точном совпадении origin/destination как в БД (ограничение поиска остаётся — это P1). Рост числа сообщений от бота; нужен лимит/антиспам позже (P1/P2).

Последствия (B):
Честный UX, меньше кода, фичу можно вернуть позже.

От вас: выбрать A (уведомлять) или B (убрать UI). По умолчанию в плане закладывается A.

## P0-4. Контакт в канале подменяется на CONTACT_USERNAME

Файлы: main.py
- env CONTACT_USERNAME;
- публикация машины (driver_add_vehicle_contact, около строки 309);
- груз пошагово (logistician_add_cargo_contact, около 455);
- parse_cargo_block / format_cargo_message / single-message confirm — контакт по умолчанию CONTACT_USERNAME, телефон из текста часто не парсится.

В БД при этом пишется message.text / переданный contact.

Почему это ошибка:
В личке/«Мои объявления» — один контакт, в канале — другой. Водители пишут посреднику, логист ждёт звонков на свой номер — срывы сделок и репутационный риск. Если подмена умышленная (агрегатор) — это должно быть явно в тексте объявления, а не скрытой заменой.

Как исправить — зафиксировать одну политику:

Вариант 1 (прозрачный агрегатор / hub):
В канале всегда CONTACT_USERNAME, но строка вида «Связь через: @… (биржа)» + в БД хранить реальный контакт для внутренних сценариев; не показывать в «Мои» один номер, а в канале другой без пояснения.

Вариант 2 (прямые контакты / user):
В channel_message / format_cargo_message подставлять введённый contact; CONTACT_USERNAME — только fallback, если пусто.

Вариант 3 (hybrid):
env PUBLISH_CONTACT_MODE=user|hub; default — ваш выбор. Для P0 достаточно одного режима без лишней абстракции.

Дополнительно для single-message: простой regex телефона (\d{9,12}, +998…) → contact, иначе fallback.

Последствия:
- user-contact: меньше контроля у владельца канала, больше прямых сделок, выше риск спама/фейков в канале;
- hub-contact: как сейчас по смыслу, но честно в тексте; нагрузка на контакт-хаб;
- сообщения в канале и логика доверия меняются — согласовать с владельцем канала.

От вас: user / hub / hybrid.

## P0-5. Скрытое уменьшение цены на 200$

Файлы: main.py
- пошаговый груз, около 440–445 (price_num - 200);
- format_cargo_message, около 615–621 (final_price = max(price_num - 200, 0)).

В БД и в ответе «груз размещен» уходит исходная цена (пошаговый путь).

Почему это ошибка:
В канале цена ниже, чем ввёл логист / чем в БД. Водитель торгуется от заниженной суммы; логист не видит расхождения в своём UI. Это либо скрытая комиссия (нужна прозрачность), либо баг.

Как исправить:

Вариант 1: убрать −200 везде; в канал и БД одна и та же цена.

Вариант 2: оставить комиссию, но:
(а) константа PRICE_CHANNEL_OFFSET = 200 + env;
(б) в превью логисту перед постом: «В канале будет показано: X−200 = Y$»;
(в) в тексте канала: «Фрахт: Y$ (условия биржи)» или две строки «Заявлено / к размещению».

Вариант 3: хранить price_raw и price_public в БД (ближе к P1 по схеме).

Для P0 хватит варианта 1 или 2 без миграции схемы.

Последствия:
- убрать −200: цены в канале вырастут на 200 к текущему поведению; старые посты не меняются;
- оставить + UI: поведение канала то же, меньше конфликтов с логистами.

Нужно согласовать с бизнес-моделью.

От вас: убрать offset / оставить, но показывать логисту в превью.

## P0-6. Неверный parse_mode при публикации груза (пошагово)

Файл: main.py, logistician_add_cargo_contact, около 446–460:
текст с *жирный* (Markdown), вызов parse_mode="HTML".
Для сравнения: машины и single-message — parse_mode="Markdown".

Почему это ошибка:
Telegram парсит HTML, а разметка Markdown: *текст* не станет жирным; при символах <, > возможны ошибки API или «сырой» вид. Несогласованность трёх путей публикации.

Как исправить:
Унифицировать. Рекомендация — везде parse_mode="Markdown" (или MarkdownV2 с корректным escape — сложнее) либо везде HTML (<b>…</b>) и переписать шаблоны.

Минимальный P0-фикс: в пошаговом грузе заменить "HTML" → "Markdown", как у остальных.

Дополнительно (желательно в том же PR): экранировать user-input в Markdown (*, _, `, [) простой функцией escape_md, чтобы города/контакты не ломали сообщение.

Последствия:
Посты грузов из пошагового режима в канале начнут отображаться с жирным/структурой как задумано. При «опасных» символах в названиях — либо починка escape, либо редкие сбои send (сейчас уже есть try/except → только log).

## P0-7. Нет валидации чисел и защита от None id

Файлы: main.py
- driver_add_vehicle_capacity / _contact (float(capacity));
- logistician_add_cargo_weight/volume/contact (float(weight/volume));
- driver_subscribe_destination (get_driver_id → add_subscription);
- driver_add_vehicle_contact (get_driver_id);
- logistician_add_cargo_contact, confirm_single_msg_cargo (get_logistician_id).

db.py — вставки с driver_id/logistician_id без проверки на уровне приложения (в схеме NOT NULL → IntegrityError).

Почему это ошибка:
«двадцать» / «20т» / пустая строка → ValueError, хендлер падает, пользователь видит молчание или generic error, state может «застрять» в середине FSM. get_*_id → None (роль не та, INSERT OR IGNORE при смене роли) → падение или битая вставка.

Как исправить:
1) Хелперы parse_positive_float(text) → float | None (замена «,»→«.», strip, > 0).
2) На шагах capacity/weight/volume: при ошибке — «Введите число, например 20» и не менять state (повтор ввода).
3) Перед add_vehicle / add_cargo / add_subscription: если id is None — «Сначала выберите роль …» + клавиатура роли / main menu, сброс state.
4) Опционально P0: try/except вокруг float на финальном шаге как страховка.
5) Single-message: не пихать «22 т» в REAL без парсинга числа — для P0: писать None/0 в weight или вытащить первое число regex’ом (минимально, без полного рефактора парсера).

Последствия:
Меньше падений бота; пользователи остаются в том же шаге FSM при опечатке. Чуть больше сообщений-подсказок. Не решает смену роли (это P1), но убирает крэши при кривых id.

## Критерий «P0 закрыт»

1. В git нет живых токенов; в проде новый BOT_TOKEN.
2. Без токена процесс не стартует с понятной ошибкой.
3. Подписки либо реально шлют сообщения, либо исчезли из UX.
4. Политика контакта и цены согласована и одинаково отражена в канале и (где нужно) в UI.
5. Пошаговый груз в канал уходит с корректной разметкой.
6. Невалидные числа и отсутствие id не роняют хендлеры.

## Архитектура проекта (кратко, для контекста)

Стек: aiogram 3, SQLite, деплой Railway.

- main.py — бот, FSM, клавиатуры, парсер объявлений, публикация в канал
- db.py — SQLite: users, cargo, vehicles, subscriptions
- requirements.txt — aiogram, aiosqlite (aiosqlite не используется)
- Procfile / railway.json — запуск python main.py
- ENV_INSTRUCTIONS.md — переменные окружения (с секретами — проблема P0-1)
- runtime.txt — Python 3.11

Роли: водитель / логист.
FSM: регистрация → меню → поиск / подписки / размещение (пошагово или одним сообщением).
